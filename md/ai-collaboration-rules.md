# AI(Gemini) 협업 규칙 가이드

본 문서는 4명의 개발자가 Gemini와 효율적으로 협업하며 코드 충돌을 방지하기 위한 규칙을 정의합니다.

## 1. 개별 파일 단위 작업 요청
*   **원칙**: 한 번에 하나의 단계(Stage) 파일만 수정을 요청하세요.
*   **예시**: "Gemini, `js/stages/stage1-chop.js` 파일에 애니메이션 효과를 추가해줘."
*   **이유**: 여러 파일을 동시에 수정하면 코드의 흐름을 놓치거나 예기치 않은 버그가 발생할 수 있습니다.

## 2. 전역 상태(AppState) 수정 금지
*   **원칙**: `js/app.js`에 정의된 `AppState`의 구조를 변경할 때는 팀원 전원의 동의가 필요합니다.
*   **이유**: `AppState`는 모든 단계에서 공유하는 상태이므로, 한 명의 수정이 다른 모든 팀원의 코드에 오류를 일으킬 수 있습니다.

## 3. 리팩토링 후 Gemini에게 검증 요청
*   **원칙**: 코드를 수정한 후에는 반드시 브라우저 도구로 결과물을 확인해달라고 Gemini에게 요청하세요.
*   **명령어 예시**: "수정된 내용을 바탕으로 Step 2가 정상적으로 작동하는지 브라우저에서 테스트해줘."

## 4. 변경 사항 요약 기능 활용
*   **원칙**: Gemini가 작업을 마친 후에는 항상 **어떤 인터페이스(함수, 변수)가 변경되었는지** 요약해달라고 하세요.
*   **이유**: 다른 팀원들이 본인의 작업 영역에 영향을 주는지 즉시 파악하기 위함입니다.

## 5. 주석 및 문서화 준수
*   **원칙**: 새로운 함수를 만들 때는 반드시 JSDoc 형태의 주석을 달도록 Gemini에게 지시하세요.
*   **지시 예시**: "모든 신규 함수에는 매개변수와 반환값에 대한 설명을 JSDoc으로 포함해줘."
